# ホテル清掃業務管理ツール 仕様書

**作成日**: 2026-05-22  
**対象ホテル**: ホテルパコジュニア 北見（6フロア・99室）  
**グループ**: パブグループ（北海道）/ グローバルビュー（全国FC）

---

## 1. 背景と目的

### 現状の運用フロー（アナログ）
1. 宿泊客がチェックアウトし、フロントにキーを返却
2. フロントが部屋番号入りのマグネットシートを清掃リーダーへ物理的に手渡す
3. リーダーが各フロアの扉にある鉄板にマグネットを貼り「清掃可能」を示す
4. 清掃スタッフが貼ってある部屋を清掃、完了後にマグネットを回収

### 現状の課題
- 複数フロアにまたがるため、清掃漏れ・未対応箇所の把握が困難
- 誰がどこを担当しているかリアルタイムに把握できない
- スタッフの出勤調整（前日に予約数を見て電話）も完全手動
- 清掃員の給与は**部屋単位の出来高制**のため、清掃ログが給与計算に直結する

### 制約事項（重要）
- ホテルの予約・入金管理システム（PMS）はグループ本部のクラウドシステムであり**改修・連携不可**
- ただし予約状況はウェブ画面で閲覧可能
- PCは本社側が設定した業務用端末のため、新たなソフトウェアの自由インストールが難しい場合がある → **ブラウザで動くWebAppが最適**

### 目的
- 清掃状況をリアルタイムで全スタッフが共有
- 未対応の部屋をひと目で確認できる
- 清掃員が自発的に空き部屋を引き受ける「手挙げ式」運用を実現
- 将来的には清掃ログを給与計算に活用できる基盤を作る

## 2. ユーザーロール

| ロール | 主な操作 | 利用デバイス |
|--------|----------|--------------|
| フロントスタッフ | チェックアウト登録・全体モニタリング | PC / タブレット |
| 清掃リーダー | 全体状況確認・検査済みマーク | スマートフォン |
| 清掃スタッフ | 自部屋の引き受け・ステータス更新 | スマートフォン |

---

## 3. 清掃タイプ（ホテルシステムから取得）

| タイプ | 略称 | 内容 |
|--------|------|------|
| チェックアウト清掃 | CO | 退室後の全清掃（リネン交換あり） |
| エコクリーニング | エコ | 連泊客の簡易清掃（リネン交換なし） |
| パス | — | 清掃不要（対応不要） |

> 現状は紙の清掃計画書（ホテルシステム出力）で管理している。COとエコの区別がスタッフへの指示として重要。

## 4. 部屋ステータスフロー

```
[使用可] → [清掃待ち (CO/エコ)] → [清掃中] → [清掃完了] → [使用可]
             ↑フロント操作          ↑清掃スタッフ  ↑清掃スタッフ  ↑リーダー操作
```

| ステータス | 色 | 操作者 | 説明 |
|-----------|-----|--------|------|
| 使用可 | 緑 | リーダー | 客室として提供できる状態 |
| 清掃待ち（CO） | 赤 | フロント | チェックアウト後の全清掃待ち |
| 清掃待ち（エコ） | オレンジ | フロント | 連泊清掃待ち（リネン交換なし） |
| 清掃中 | 黄 | 清掃スタッフ | 担当者が手挙げで引き受け中 |
| 清掃完了 | 青 | 清掃スタッフ | 清掃完了・リーダー確認待ち |

## 4. 機能要件

### 4-1. フロアマップ表示（全ロール共通）
- フロア別のグリッド表示（タブ or スクロールで切り替え）
- 各部屋をカード形式で表示（部屋番号・ステータス色・担当者名）
- 「全フロア一覧」ビューと「フロア別詳細」ビューを切り替え可能
- 未清掃（清掃待ち）の部屋数をフロアヘッダーにバッジ表示

### 4-2. チェックアウト登録（フロント）
- 部屋番号をタップ → 「チェックアウト済にする」ボタンで即時更新
- 複数部屋を一括でチェックアウト済にするバルク操作
- 更新時刻を自動記録

### 4-3. 手挙げ式清掃引き受け（清掃スタッフ）
- 「清掃待ち」の部屋をタップ → 「私が担当します」ボタン
- 担当後はステータスが「清掃中」へ変わり、担当者名が表示される
- 「清掃完了」ボタンで完了報告
- 一度引き受けた部屋は他スタッフに表示上でロック（二重担当防止）

### 4-4. 検査・使用可確定（リーダー）
- 「清掃完了」の部屋を確認後「検査済み/使用可」にマーク
- 検査時刻を自動記録

### 4-5. リアルタイム同期
- WebSocket（またはServer-Sent Events）で全端末にリアルタイム反映
- ページリロード不要

### 4-6. プッシュ通知（将来拡張・初期設計に含める）
- チェックアウトが入ったらリーダーへ通知
- 清掃完了が入ったらリーダーへ通知
- PWA（Progressive Web App）として実装し、ホーム画面追加で擬似ネイティブ化

### 4-7. ダッシュボード（フロント・リーダー向け）
- 本日の清掃進捗サマリー（完了室数 / 総対象室数）
- フロア別の完了率バー
- 清掃スタッフ別の対応件数

## 7. 画面構成（案）

```
/ (ログイン)
├── /dashboard        ← フロント・リーダー向けサマリー
├── /floors           ← 全フロアマップ（メイン画面）
│   └── /floors/:id  ← フロア別詳細
├── /rooms/:id        ← 部屋詳細・ステータス更新
├── /admin
│   ├── /admin/rooms  ← 部屋マスタ管理
│   └── /admin/users  ← スタッフアカウント管理
└── /history          ← 操作ログ・完了履歴
```

## 8. 技術スタック（案）

| レイヤー | 採用候補 | 理由 |
|---------|---------|------|
| フロントエンド | React + Vite | PWA対応、モバイル最適化しやすい |
| バックエンド | Node.js (Express) または FastAPI (Python) | 軽量・WebSocket対応 |
| DB | PostgreSQL (Supabase) | リアルタイム機能内蔵、無料枠あり |
| リアルタイム | Supabase Realtime or Socket.io | フロアマップの即時反映 |
| ホスティング | Vercel (フロント) + Supabase (BaaS) | 低コスト・運用が楽 |
| プッシュ通知 | Web Push API (PWA) | 追加アプリ不要 |

## 11. 将来的なビジネス展開メモ

- 北見エリアには旅館・ホテル組合があり、地場系施設は同様の課題を抱えている
- フランチャイズ系（東横インなど）はシステムが整っているが、**地場系は手つかず**
- 現場で作ったツールは他施設への横展開・貸し出しポテンシャルがある
- 竹内さん（IoT専門）との協力で音・混雑センサーなど物理層との連携も視野に入れられる
