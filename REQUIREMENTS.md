# ホテルパコジュニア 北見 — 清掃管理システム 要件定義

> 最終更新: 2026-05-22  
> ステータス: 実装中（一部機能は未着手）

---

## 1. システム概要

ホテルパコジュニア 北見（6フロア・99室）向けの清掃管理 Web アプリ。  
スタッフが個人スマートフォンでアクセスし、当日の清掃状況をリアルタイムで共有する。  
管理者（リーダー/フロント）は清掃計画の作成・担当割り当て・進捗確認を行う。

---

## 2. ユーザー種別と権限

| ロール | 説明 | アクセス URL |
|--------|------|-------------|
| `cleaner`（清掃スタッフ） | 部屋の担当開始・清掃完了記録 | `/` |
| `leader`（リーダー） | 計画作成・割り当て・承認・全機能 | `/admin` |
| `front`（フロント） | チェックアウト登録・進捗確認 | （今後検討） |

### 認証仕様（現状）
- URL ベースのモード切り替え（`/` = スタッフ、`/admin` = 管理者）
- ハードコードのテスト認証情報（Supabase Auth は未使用）
  - スタッフ: `staff / 1234` → ログイン後に名前選択
  - 管理者: `admin / admin` → 固定名「管理者」
- セッションは localStorage に保存（モード別に別キー）

### 将来対応
- スタッフ個人ごとのアカウントを作成し、担当割り当てを自動化

## 4. 通常清掃（Floors.jsx）

### 部屋ステータスの遷移

```
stay（在室中/グレー）
  │ フロントがチェックアウト登録
  ▼
checkout（清掃待ち/赤 or オレンジ）
  │ スタッフが「担当する」タップ
  ▼
cleaning（清掃中/アンバー）
  │ スタッフがアメニティ記録して「完了」
  ▼
cleaned（確認待ち/ブルー）
  │ リーダーが「検査完了」承認
  ▼
available（清掃済み/グリーン）
```

- `stay` はデフォルト（ゲスト在室、清掃不要）
- `available` は清掃・検査完了、次のゲストを迎えられる状態

### 清掃種別

| 種別 | 表示 | 条件 |
|------|------|------|
| CO清掃 | 赤バッジ | 当日チェックアウト（全室作業） |
| エコ清掃 | オレンジバッジ | 長期滞在中の定期清掃（3泊ごと） |
| 追加清掃 | 紫バッジ | 当日追加登録分 |

### フロア構成（マスター）

| フロア | 部屋 |
|--------|------|
| 2F | 201(TR), 202(T), 203(W), 205-211(S) |
| 3-7F | X01(W), X02(W), X03(W), X05-X21(S/T/W) ※パターンあり |

### 部屋タイプ

| タイプ | 説明 | 清掃ポイント |
|--------|------|-------------|
| S | シングル | 1.0 pt |
| SD | セミダブル | 1.0 pt |
| W | ダブル/ワイド | 1.0 pt |
| T | ツイン | 1.2 pt |
| TR | トリプル | 2.0 pt |

> 部屋タイプは現地の変更に応じて管理者が変更できる必要がある（→ 部屋マスター管理機能）

### アメニティ記録

清掃完了時にスタッフが補充したアメニティ数を入力：

- バスタオル / フェイスタオル / ウォッシュタオル / バスマット
- アメニティセット / シャンプー / ボディソープ / ティッシュ

CO清掃はデフォルト各1個、エコ清掃は消耗品（シャンプー/ボディソープ）のみ補充。

## 5. 翌日計画（CleaningPlan.jsx）

### Excel 取り込み仕様

- PMS（予約管理システム）から書き出した XLS/XLSM ファイルをアップロード
- 解析対象列: 列 D（泊数, index=3）/ 列 F（部屋番号, index=5）
- ヘッダー行: 3 行（4 行目からデータ）

### 清掃種別の判定ロジック

```
泊数フィールド形式: "現在の泊目/総泊数"（例: "3/6" = 6泊のうち3泊目の夜）

current == total         → CO清掃（チェックアウト前夜）
total > 5 AND current % 3 == 0 → エコ清掃（長期滞在の3泊ごと）
それ以外                 → null（清掃なし）
```

### 自動割り当てロジック

- 各スタッフに「ポイント上限（target）」を設定
- 2F → 7F の昇順に部屋を並べ、ポイントを消費しながら順に割り当て
- 1人のポイント上限を超えたら次のスタッフへ
- 残りの部屋は最後のスタッフに割り当て（オーバーフロー）
- 動線を最小化するため、各スタッフは連続したフロアを担当

### アラート（警告表示）

| 条件 | レベル | メッセージ |
|------|--------|-----------|
| 割り当て不能な部屋がある | error | 未割り当て部屋一覧を表示 |
| 総ポイントが上限合計を超過 | warn | 超過ポイント数を表示 |
| 総ポイントが上限の 60% 未満 | info | スタッフ削減を提案 |

### 計画の保存・履歴（未実装）

- 翌日の清掃計画を「確定」するとその日の日付キーで保存
- 過去の計画はカレンダー画面から参照できる
- 計画が保存されていない日はカレンダー上で「要設定」として色分け表示
- メンテナンス日・定休日をカレンダー上でマーク可能（清掃不要の日）

## 6. 当日追加（ExtraCleanings.jsx）

### 追加清掃登録機能

- 清掃種別を選択: 追加清掃（デフォルト）/ CO清掃 / エコ清掃
- 部屋番号を数字入力（3桁）で登録
- 「部屋を選んで追加」→ フロア別部屋一覧をポップアップ表示
  - 全フロア/各フロア切り替えタブ
  - タップで部屋を選択（複数選択可）
  - 追加済み部屋は色塗り＋「追加済」ラベルで視覚化
- 同じ部屋+種別の組み合わせは重複登録しない
- メモ欄あり（任意）

---

## 7. スタッフ管理（Staff.jsx）

- スタッフの名前・目標ポイント（target）を編集
- 出勤 ON/OFF トグル（OFF スタッフは割り当て対象外）
- 追加・削除対応
- localStorage に保存（`hotel_staff_config`）
- デフォルト: 結城(11pt)・戸田(10pt)・森山(10pt)

---

## 8. 部屋マスター管理（未実装）

### 目的

部屋タイプ（シングル/セミダブル/ダブル/ツイン/トリプル）は変更されることがあるため、管理者が設定変更できる画面が必要。

### 機能要件

- 全部屋一覧をフロア別に表示
- 各部屋の「部屋タイプ」を変更可能（S / SD / W / T / TR）
- 変更時に「変更者の名前」を入力・申告してもらう（ユーザー認証が個人単位でないため）
- 変更履歴を記録: 変更日時・変更前タイプ・変更後タイプ・変更者名
- 変更履歴は一覧表示できる

### 保存先

- localStorage（オフライン対応）または Supabase の `rooms` テーブル

---

## 9. カレンダー機能（未実装）

### 目的

翌日の清掃計画を日別で管理し、過去の記録を確認できるようにする。

### 機能要件

- カレンダー UI（月表示）で各日のステータスを色分け
  - 🟢 確定済み（清掃計画が保存・確定されている）
  - 🟡 未設定（計画が入力されていない将来の日）
  - ⬜ 定休日・メンテナンス日（手動でマーク）
  - ⬛ 過去日（参照のみ）
- 日付をタップ → その日の清掃計画詳細を表示
- 翌日の計画を「確定」すると保存（上書き確認あり）
- 定休日・メンテナンス日をカレンダー上でマーク/解除

### 保存先

- localStorage（`hotel_cleaning_plans`）に日付キーで JSON 保存

## 10. 技術スタック

| 項目 | 採用技術 |
|------|---------|
| フロントエンド | React 18 + Vite |
| スタイリング | Tailwind CSS v3 |
| Excel 解析 | SheetJS (xlsx v0.18.5) |
| リアルタイム DB | Supabase（Postgres + Realtime） |
| ローカル状態 | localStorage |
| ホスティング | Vercel（SPA rewrites） |

### Supabase テーブル構成

```sql
-- 部屋マスター
rooms (
  id           TEXT PRIMARY KEY,  -- 部屋番号 (例: "301")
  floor        INT,
  room_number  TEXT,
  room_type    TEXT,              -- S / SD / W / T / TR
  created_at   TIMESTAMPTZ,
  updated_at   TIMESTAMPTZ
)

-- 当日の清掃ステータス
room_status (
  id              TEXT PRIMARY KEY REFERENCES rooms(id),
  status          TEXT,  -- stay / checkout / cleaning / cleaned / available
  cleaning_type   TEXT,  -- co / eco / extra / null
  assigned_staff  TEXT,
  checkout_at     TIMESTAMPTZ,
  cleaning_start_at TIMESTAMPTZ,
  cleaned_at      TIMESTAMPTZ,
  amenities       JSONB,
  updated_at      TIMESTAMPTZ,
  updated_by      TEXT
)
```
